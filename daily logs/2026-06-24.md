## Workstream B — Performance & Caching Architecture

### 3. Middleware / Proxy Consolidation
- **Files:** `proxy.ts` (rewritten), `middleware.ts` (deleted)
- **Problem:** Next.js 16 detected both `middleware.ts` and `proxy.ts`, which is not allowed ("Both middleware file and proxy file are detected").
- **Solution:** Consolidated all logic into `proxy.ts`, deleted `middleware.ts`. The new proxy handles three concerns:
  - **Admin route protection** — Checks `x-admin-key` header or `admin_key` cookie against `ADMIN_ACCESS_KEY` env var for `/admin/*` and `/api/admin/*` paths; returns 401 JSON for unauthorized API requests.
  - **Rate limiting** — Per-IP tracking via `x-forwarded-for` / `x-real-ip`; 60 requests per 60-second window for all `/api/*` routes (excluding admin); returns 429 with `Retry-After` header when exceeded; stale entries cleaned when map exceeds 1000 entries.
  - **Cache headers & telemetry** — Sets `Cache-Control: public, s-maxage=60, stale-while-revalidate=300` on all GET API routes; adds `X-Response-Time` header (millisecond precision) to every response for diagnostics.
- **Matcher config:** `['/admin/:path*', '/api/admin/:path*', '/api/:path*']`

### 4. Chat API — AI Resilience & Caching
- **File:** `app/api/chat/route.ts` (full rewrite)
- **Key changes:**
  - **AbortController timeout:** External Groq API fetch now aborts after 25 seconds (`TIMEOUT_MS = 25_000`). Prevents hanging when Groq is slow or unresponsive.
  - **Response cache:** In-memory `Map<string, { data, ts }>` caches the last 3 messages for 10 seconds (`CACHE_TTL = 10_000`). Deduplicates identical user messages (e.g., rapid double-clicks) and avoids redundant API calls.
  - **Smart model fallback:** First attempt uses `llama-3.3-70b-versatile` (full model); retries use `llama-3.1-8b-instant` (8× faster, lower latency). This provides best quality on first try but faster recovery on failure.
  - **Trimmed system prompt:** Reduced from ~160 lines to ~30 lines. Removed verbose tone instructions and repeated examples — the core rules, sitemap, and product info are preserved. Fewer tokens per request means faster response and lower cost.
  - **Input validation:** Early return 400 if `messages` is missing or empty.
  - **Tightened error handling:** Only retries on 401 (unauthorized) or 429 (rate limit). Other errors still retry but with clearer logging.

### 5. Search API — AI Result Caching & Timeouts
- **File:** `app/api/search/route.ts` (full rewrite)
- **Key changes:**
  - **AI response cache:** Both `getAIRecommendations()` and `generateSearchSummary()` are wrapped in `getCachedOrFetch()` with a 5-minute TTL (`SUMMARY_CACHE_TTL = 300_000`). Same query from different users within 5 minutes reuses the cached AI response.
  - **AbortController timeouts:** Every external Groq fetch in search now uses `fetchWithTimeout()` with an 8-second limit for recommendations and summaries (these are low-criticality features).
  - **Reduced AI dependency:** AI fallback recommendations only trigger when combined results < 2 (was < 3). This means more searches resolve without any AI call.
  - **Faster model:** Both recommendations and summaries now use `llama-3.1-8b-instant` instead of `llama-3.3-70b-versatile` — these tasks don't need the full model's reasoning depth.
  - **Static page matching without DB:** Extracted `STATIC_PAGES` and `matchStaticPages()` to return instant results for common search terms (products, services, etc.) without hitting MongoDB.
  - **Deduplication helper:** Centralized `deduplicate()` function to avoid repeated `seen` set logic.
  - **JSON response format:** AI recommendations use `response_format: { type: "json_object" }` for reliable parsing.

### 6. Cache-Control Headers — Settings & Content APIs
- **Files:** `app/api/settings/route.ts`, `app/api/content/route.ts`
- **Settings API:** Added `Cache-Control: public, s-maxage=300, stale-while-revalidate=600` (5 min CDN cache, 10 min stale revalidation). Settings change rarely, so a 5-minute cache dramatically reduces load.
- **Content API:** Added `Cache-Control: public, s-maxage=120, stale-while-revalidate=300` (2 min CDN cache, 5 min stale). Content is edited via admin panel, so 2 min is a good balance of freshness vs. load reduction.

### 7. CDN & Static Asset Caching
- **File:** `next.config.ts`
- **Added header rules:**
  - **Static assets** (`.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`, `.avif`, `.svg`, `.ico`, `.css`, `.js`, `.woff2`): `Cache-Control: public, max-age=31536000, immutable` — 1-year cache with immutable directive; these are content-hashed by Next.js.
  - **Other non-HTML files:** `Cache-Control: public, max-age=86400, stale-while-revalidate=604800` — 1-day cache, 1-week stale revalidation for anything else that isn't HTML.
- **Why:** Before this change, static assets had no explicit cache headers from Next.js. CDNs and browsers would revalidate on every visit. These headers eliminate unnecessary revalidation for versioned assets and dramatically reduce origin load.

### 8. MongoDB Connection Tuning
- **File:** `lib/mongodb.ts`
- **Changes:**
  - `connectTimeoutMS`: 10,000 → **5,000** (connect failure detection 2× faster)
  - `socketTimeoutMS`: 45,000 → **30,000** (socket hang detection 1.5× faster)
  - `maxPoolSize`: 10 → **25** (2.5× more concurrent database operations)
  - Added `serverSelectionTimeoutMS: 5,000` (was implicit — now explicit for faster failover)
  - Added `heartbeatFrequencyMS: 10,000` (monitor server health every 10s instead of default)
- **Why:** The previous pool of 10 connections could become a bottleneck during traffic spikes (especially with the Chat API holding connections while waiting for Groq). 25 connections with faster timeouts provides better throughput under load.

### 9. NewsFeed API Routing Bugfix
- **Files:** `app/components/NewsFeed.tsx`, `app/api/news/route.ts` (NEW)
- **Problem:** `NewsFeed.tsx` (a public client component rendered in the site layout) was fetching from `/api/admin/news`, an admin-protected endpoint. When `ADMIN_ACCESS_KEY` was configured, the proxy returned a 401 JSON. When not configured, it worked accidentally. But more critically, the component showed `SyntaxError: Unexpected token '<', "<!DOCTYPE "...` — meaning the browser received an HTML page instead of JSON, likely because the admin API route fell through to Next.js's error page in certain environments.
- **Solution:** Created a public `/api/news` GET endpoint at `app/api/news/route.ts` that calls `getNews()` and returns JSON with `Cache-Control: public, s-maxage=120, stale-while-revalidate=300`. Updated NewsFeed to fetch from `/api/news` instead of `/api/admin/news`.
- **New file:** `app/api/news/route.ts` — 16 lines, simple passthrough to `getNews()` with cache headers.

### 10. Search Page — Misleading "No Exact Match" Note
- **File:** `app/search/page.tsx` (line ~134)
- **Problem:** The condition `results.some(r => r.title.toLowerCase() === query.toLowerCase())` required a result's **title to exactly equal** the query. This was too strict — searching "TallyPrime" would find "TallyPrime Silver", "TallyPrime Gold", etc., but the exact equality check would fail because the titles contained additional words. Users would see the amber warning "Note: I cannot find an exact match, but here are the most relevant results" even when results directly contained the query term.
- **Fix:** Replaced with a three-pronged `hasGoodMatch` check:
  1. `r.title.toLowerCase().includes(q)` — query is a substring of the title
  2. `r.title.toLowerCase().split(/\s+/).some(w => w === q)` — query is a whole word in the title
  3. `r.description?.toLowerCase().includes(q)` — query appears in the description
- **Result:** The warning note now only shows when there is genuinely no close match in any result title or description.
